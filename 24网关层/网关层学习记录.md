# 网关层学习记录

## 项目概述

这是一个基于 **Fastify** 框架构建的 API 网关项目，实现了微服务架构中网关层的核心功能。

## 技术栈

- **框架**: Fastify (高性能 Node.js Web 框架)
- **核心依赖**:
  - `@fastify/http-proxy` - 负载代理
  - `@fastify/rate-limit` - 限流
  - `@fastify/caching` - 缓存
  - `opossum` - 熔断器

## 核心功能实现

### 1. 反向代理 (Reverse Proxy)

**配置文件**: `src/proxy/index.js`

实现了两个服务的代理转发：
- `/pc` 路径 → 转发到 `http://localhost:9001`
- `/mobile` 路径 → 转发到 `http://localhost:9002`

支持的 HTTP 方法：GET, POST, PUT, DELETE

### 2. 限流技术 (Rate Limiting)

**配置**: `src/config/index.js` - `rateLimitConfig`

```javascript
{
  max: 5,              // 最大请求次数
  timeWindow: 60000    // 时间窗口：60秒
}
```

**作用**: 防止单个客户端在 60 秒内发送超过 5 个请求，保护后端服务不被过载。

### 3. 缓存技术 (Caching)

**配置**: `src/config/index.js` - `cachingConfig`

```javascript
{
  max: 1000,           // 最大缓存条目数
  expire: 60000        // 过期时间：60秒
}
```

**作用**: 缓存响应结果，减少对后端服务的重复请求，提升响应速度。

### 4. 熔断技术 (Circuit Breaker)

**配置**: `src/config/index.js` - `breakerConfig`

```javascript
{
  timeout: 6000,           // 超时时间：6秒
  errorThreshold: 50,      // 错误率阈值：50%
  resetWindow: 6000        // 重置窗口：6秒
}
```

**实现原理**:
- 使用 `opossum` 库实现熔断器模式
- 在代理请求前通过 `preHandler` 检测目标服务健康状态
- 当错误率超过 50% 时，熔断器打开，直接返回 503 错误
- 6 秒后尝试重置，检查服务是否恢复

**熔断响应**:
```javascript
{
  code: 503,
  msg: '熔断器已打开，服务不可用，请稍后重试。'
}
```

## 启动方式

网关服务运行在 **3000 端口**：
```bash
node src/index.js
```

测试服务器（可指定端口）：
```bash
node server.js 9001
node server.js 9002
```

## 开发思路

### 第一步：搭建基础代理
1. 选择 Fastify 框架（性能优于 Express）
2. 安装 `@fastify/http-proxy` 实现反向代理
3. 配置代理规则，将不同路径转发到不同的后端服务
4. 测试基本的请求转发功能

### 第二步：添加限流保护
1. 安装 `@fastify/rate-limit` 插件
2. 配置限流参数（时间窗口、最大请求数）
3. 通过 `app.register()` 注册限流中间件
4. 测试限流效果，验证超出限制时的响应

### 第三步：引入缓存机制
1. 安装 `@fastify/caching` 插件
2. 配置缓存策略（最大条目数、过期时间）
3. 注册缓存中间件，自动缓存 GET 请求
4. 验证缓存命中率和性能提升

### 第四步：实现熔断保护
1. 安装 `opossum` 熔断器库
2. 创建熔断器实例，配置超时、错误阈值、重置窗口
3. 在代理的 `preHandler` 钩子中检测服务健康状态
4. 当服务不可用时，熔断器打开，快速返回 503 错误
5. 测试服务宕机场景，验证熔断器行为

### 第五步：模块化配置
1. 将代理配置抽离到 `src/proxy/index.js`
2. 将限流、缓存、熔断配置抽离到 `src/config/index.js`
3. 保持主文件 `src/index.js` 简洁清晰
4. 便于后续维护和扩展

### 开发顺序建议
```
基础代理 → 限流 → 缓存 → 熔断 → 配置优化
```

每一步都应该独立测试，确保功能正常后再进行下一步。

## 学习要点

### 网关层的核心职责
1. **路由转发**: 根据请求路径将流量分发到不同的微服务
2. **流量控制**: 通过限流保护后端服务
3. **性能优化**: 通过缓存减少后端压力
4. **容错处理**: 通过熔断器防止故障扩散

### 设计模式
- **熔断器模式 (Circuit Breaker)**: 当后端服务出现故障时，快速失败而不是持续等待
- **代理模式 (Proxy)**: 网关作为客户端和服务端之间的中间层

### 关键技术点
1. **Fastify 插件系统**: 通过 `register()` 方法注册中间件
2. **preHandler 钩子**: 在请求处理前执行自定义逻辑
3. **熔断器状态机**: 关闭 → 打开 → 半开 → 关闭
4. **限流算法**: 滑动窗口算法控制请求频率

### 可优化的方向
1. 添加日志记录功能
2. 实现负载均衡（多个后端实例）
3. 添加认证和鉴权中间件
4. 实现更复杂的路由规则
5. 添加监控和指标收集
6. 实现请求重试机制
